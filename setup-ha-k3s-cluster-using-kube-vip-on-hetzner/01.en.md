## Introduction

This tutorial is a continuation of the
[K3s cluster with autoscaling on Hetzner](https://community.hetzner.com/tutorials/k3s-cluster-with-autoscaling)
tutorial. In that guide we created a K3s cluster with autoscaling worker nodes.
Here we will make **control plane highly available** using three master nodes
and a Virtual IP (VIP), without adding a Hetzner Load Balancer in front of it.

Standard approach for HA K8s control planes on cloud providers is to place a load
balancer in front of Kube API server. If you don't want to pay for an extra load
balancer or simply want to explore an open-source alternative,
[Kube-VIP](https://kube-vip.io/) is worth looking at.

**Challenge on Hetzner (and every major cloud provider):** Kube-VIP's default
ARP mode works by broadcasting a Gratuitous ARP packet when leader changes,
which causes every host on the LAN to update its IP-to-MAC mapping for the VIP.
Hetzner Cloud, like all large cloud providers, runs its networking at Layer 3 via
a Software Defined Network (SDN). VMs do not share an Ethernet broadcast domain,
so those ARP broadcasts are silently dropped by the network gateway and the VIP
never actually moves.

**Workaround:** We use Kube-VIP only for its **Kubernetes Lease-based leader
election**. K3s holds an election among control plane nodes to decide who takes over
managing the cluster. When a node wins the election, Kube-VIP binds the VIP address to its
private network interface locally. A lightweight `systemd` service on each master
polls for this binding every five seconds. As soon as it detects that this node is
now leader, it calls Hetzner API to update a **private network route**
(`10.10.10.10/32`) to point to the new leader server. From that moment on, any
traffic to `10.10.10.10` inside Hetzner private network is forwarded to the
correct master — no ARP required.

**What you will have at the end of this tutorial:**

- Three K3s master (control-plane) nodes joined in an HA cluster
- Kube-VIP running as a DaemonSet on all three masters for leader election
- `hetzner-vip-sync` systemd service on every master that updates Hetzner
  private network route on failover
- A stable private VIP `10.10.10.10` that worker nodes and your kubeconfig
  always connect to

### Prerequisites

- A Hetzner Cloud private network already configured. If you have not done this
  yet, follow **Step 1 of the
  [K3s cluster with autoscaling](https://community.hetzner.com/tutorials/deploy-k3s-on-hetzner-with-autoscaling)
  tutorial** to create the network, then return here.

  > **Important — start with fresh servers.** K3s API server generates its
  > TLS certificate once at cluster init time. The `--tls-san` flag (which makes
  > the certificate valid for the VIP address) must be present at that first boot.
  > If you reuse master nodes from a previous tutorial that were not bootstrapped
  > with `--tls-san=10.10.10.10`, the certificate will not cover the VIP and
  > `kubectl` connections to it will fail.
  > **Do not reuse existing master nodes** — servers in this tutorial are all
  > created fresh in Step 3.

- Hetzner Cloud
  [API token](https://docs.hetzner.com/cloud/api/getting-started/generating-api-token)
  with **Read & Write** permissions
- The [Hetzner CLI (`hcloud`)](https://github.com/hetznercloud/cli) installed and
  authenticated on your local machine
- Basic understanding of `kubectl` and `systemd`

## Step 1 - Understand Why ARP-based VIPs Do Not Work on Hetzner

Before deploying anything it helps to understand the core problem and why our
workaround is necessary.

### Step 1.1 - How Kube-VIP normally works on bare metal

On a traditional bare-metal or on-premises Layer 2 network, all VMs share
the same Ethernet broadcast domain. When Kube-VIP elects a new leader it sends a
**Gratuitous ARP** frame: *"IP `10.10.10.10` is now at MAC address
`aa:bb:cc:dd:ee:ff`"*. Every host on the segment receives this broadcast and
immediately updates its ARP cache. Traffic to the VIP starts flowing to the new
leader within milliseconds.

Here's the
[Kube-VIP ARP mode overview](https://kube-vip.io/docs/modes/arp/).

### Step 1.2 - Why it fails on Hetzner Cloud

Hetzner Cloud's private network
([architecture docs](https://docs.hetzner.com/networking/networks/technical-concepts/architecture/))
is a **Layer 3 SDN**. It is a strictly controlled routing environment where servers are isolated and cannot broadcast messages to each other.
This means:

- ARP broadcast frames are **not forwarded** between VMs by the gateway.
- Gratuitous ARP packets from Kube-VIP are dropped.

### Step 1.3 - Hetzner Network Route approach

Hetzner's private network allows us to assign an Alias IP to any server,
which can be dynamically reassigned at any time via Hetzner Cloud API.
By telling the API to attach our Virtual IP (10.10.10.10) to a specific Server ID,
Hetzner's central SDN instantly updates its internal routing table to send traffic to the new master node.

To automate this, our custom ``hetzner-vip-sync`` service acts as the bridge.
It constantly watches the local network interface, and whenever it detects that Kube-VIP has elected the local node as the new leader, the script instantly fires the API call to reassign the Alias IP. The entire failover takes effect within seconds.

## Step 2 - Get the Private Network ID

If you have not created a private network yet, follow **Step 1 of the
[K3s cluster with autoscaling](https://community.hetzner.com/tutorials/deploy-k3s-on-hetzner-with-autoscaling)
tutorial** to create one, then return here.

Once your network is created, fetch its numeric ID using Hetzner API:

```bash
curl -s -H "Authorization: Bearer <your_hcloud_token>" \
  "https://api.hetzner.cloud/v1/networks" | grep -A 1 '"id"'
```

Note down the numeric ID from the output (e.g. `12169571`). You will need
this value in Step 3 when configuring the VIP sync service.

## Step 3 - Create the Master Nodes with Cloud-Init

We will create all three master nodes using cloud-init so that each server
configures itself on first boot: it installs K3s with the correct flags, deploys
the Kube-VIP manifest, and starts the `hetzner-vip-sync` service automatically.

### Step 3.1 - Prepare the cloud-init file for master node 1

Master node 1 initialises the etcd cluster with `--cluster-init`. Create a file
called `master-1-init.yaml` on your local machine:

```yaml
#cloud-config

write_files:
  # Kube-VIP DaemonSet manifest
  - path: /var/lib/rancher/k3s/server/manifests/kube-vip.yaml
    permissions: "0644"
    content: |
      apiVersion: v1
      kind: ServiceAccount
      metadata:
        name: kube-vip
        namespace: kube-system
      ---
      apiVersion: rbac.authorization.k8s.io/v1
      kind: ClusterRole
      metadata:
        annotations:
          rbac.authorization.kubernetes.io/autoupdate: "true"
        name: system:kube-vip-role
      rules:
        - apiGroups: [""]
          resources: ["services", "services/status", "nodes", "endpoints"]
          verbs: ["list", "get", "watch", "update"]
        - apiGroups: ["coordination.k8s.io"]
          resources: ["leases"]
          verbs: ["list", "get", "watch", "update", "create"]
      ---
      kind: ClusterRoleBinding
      apiVersion: rbac.authorization.k8s.io/v1
      metadata:
        name: system:kube-vip-binding
      roleRef:
        apiGroup: rbac.authorization.k8s.io
        kind: ClusterRole
        name: system:kube-vip-role
      subjects:
      - kind: ServiceAccount
        name: kube-vip
        namespace: kube-system
      ---
      apiVersion: apps/v1
      kind: DaemonSet
      metadata:
        name: kube-vip-ds
        namespace: kube-system
      spec:
        selector:
          matchLabels:
            name: kube-vip-ds
        template:
          metadata:
            labels:
              name: kube-vip-ds
          spec:
            affinity:
              nodeAffinity:
                requiredDuringSchedulingIgnoredDuringExecution:
                  nodeSelectorTerms:
                  - matchExpressions:
                    - key: node-role.kubernetes.io/master
                      operator: Exists
                  - matchExpressions:
                    - key: node-role.kubernetes.io/control-plane
                      operator: Exists
            serviceAccountName: kube-vip
            hostNetwork: true
            containers:
            - args:
              - manager
              env:
              - name: vip_arp
                value: "true"   # No effect on Hetzner; election still works
              - name: port
                value: "6443"
              - name: nodename
                valueFrom:
                  fieldRef:
                    fieldPath: spec.nodeName
              - name: vip_interface
                value: "enp7s0"
              - name: vip_cidr
                value: "32"
              - name: cp_enable
                value: "true"
              - name: cp_namespace
                value: kube-system
              - name: vip_ddns
                value: "false"
              - name: svc_enable
                value: "false"
              - name: vip_leaderelection
                value: "true"
              - name: address
                value: "10.10.10.10"
              image: ghcr.io/kube-vip/kube-vip:v0.8.9
              imagePullPolicy: IfNotPresent
              name: kube-vip
              securityContext:
                capabilities:
                  add:
                  - NET_ADMIN
                  - NET_RAW
                  - SYS_TIME
            tolerations:
            - effect: NoSchedule
              operator: Exists
            - effect: NoExecute
              operator: Exists

  # Hetzner VIP sync script
  - path: /usr/local/bin/hetzner-vip-sync.sh
    permissions: "0755"
    content: |
      #!/bin/bash
      PRIVATE_VIP="10.10.10.10"
      INTERFACE="enp7s0"
      HCLOUD_TOKEN="<your_hcloud_token>"
      NETWORK_ID="<your_network_id>"
      SERVER_ID=$(curl -sf \
        http://169.254.169.254/hetzner/v1/metadata/instance-id)
      WAS_LEADER=false
      echo "Starting Hetzner VIP Sync -- Server ID: ${SERVER_ID}"
      while true; do
          if ip -4 addr show dev "${INTERFACE}" | grep -q "${PRIVATE_VIP}"; then
              IS_LEADER=true
          else
              IS_LEADER=false
          fi
          
          if [ "${IS_LEADER}" = true ] && [ "${WAS_LEADER}" = false ]; then
              echo "$(date): Became leader. Updating Hetzner Alias IP..."
              curl -sf -X POST \
                "https://api.hetzner.cloud/v1/servers/${SERVER_ID}/actions/change_alias_ips" \
                -H "Authorization: Bearer ${HCLOUD_TOKEN}" \
                -H "Content-Type: application/json" \
                -d "{\"network\": ${NETWORK_ID}, \"alias_ips\": [\"${PRIVATE_VIP}\"]}"
              WAS_LEADER=true
          elif [ "${IS_LEADER}" = false ] && [ "${WAS_LEADER}" = true ]; then
              echo "$(date): Lost leadership."
              WAS_LEADER=false
          fi
          sleep 5
      done

  # systemd unit for the sync service
  - path: /etc/systemd/system/hetzner-vip-sync.service
    content: |
      [Unit]
      Description=Hetzner VIP Sync for Kube-VIP
      After=network-online.target
      Wants=network-online.target

      [Service]
      Type=simple
      ExecStart=/usr/local/bin/hetzner-vip-sync.sh
      Restart=always
      RestartSec=10
      User=root

      [Install]
      WantedBy=multi-user.target

runcmd:
  - systemctl daemon-reload
  - systemctl enable hetzner-vip-sync.service
  - |
    curl -sfL https://get.k3s.io | \
      INSTALL_K3S_VERSION="v1.35.4+k3s1" \
      K3S_TOKEN="<your_k3s_token>" \
      INSTALL_K3S_EXEC="server \
        --cluster-init \
        --disable-cloud-controller \
        --disable=servicelb \
        --disable=traefik \
        --kubelet-arg=cloud-provider=external \
        --node-ip=10.10.10.11 \
        --flannel-iface=enp7s0 \
        --tls-san=10.10.10.10 \
        --tls-san=10.10.10.11 \
        --node-label node-type=master" \
      sh -
  - systemctl start hetzner-vip-sync.service
```

> **Before using this file:** Replace `<your_hcloud_token>`,
> `<your_network_id>`, and `<your_k3s_token>` with your real values.
> Always check
> [kube-vip releases](https://github.com/kube-vip/kube-vip/releases)
> for the latest stable image tag before deploying.

### Step 3.2 - Prepare the cloud-init files for master nodes 2 and 3

Master nodes 2 and 3 join the existing etcd cluster by pointing `--server` at
master node 1's private IP. Kube-VIP manifest, sync script, and systemd
unit are identical to master 1 — only the K3s install command differs.

Create `master-2-init.yaml`. The `write_files` section is the same as in
`master-1-init.yaml`. Only the `runcmd` K3s install block changes:

```yaml
runcmd:
  - systemctl daemon-reload
  - systemctl enable hetzner-vip-sync.service
  - |
    curl -sfL https://get.k3s.io | \
      INSTALL_K3S_VERSION="v1.35.4+k3s1" \
      K3S_TOKEN="<your_k3s_token>" \
      INSTALL_K3S_EXEC="server \
        --server https://10.10.10.11:6443 \
        --disable-cloud-controller \
        --disable=servicelb \
        --disable=traefik \
        --kubelet-arg=cloud-provider=external \
        --node-ip=10.10.10.12 \
        --flannel-iface=enp7s0 \
        --node-label node-type=master" \
      sh -
  - systemctl start hetzner-vip-sync.service
```

Create `master-3-init.yaml` — identical to `master-2-init.yaml` with
`--node-ip=10.10.10.13`:

```yaml
runcmd:
  - systemctl daemon-reload
  - systemctl enable hetzner-vip-sync.service
  - |
    curl -sfL https://get.k3s.io | \
      INSTALL_K3S_VERSION="v1.35.4+k3s1" \
      K3S_TOKEN="<your_k3s_token>" \
      INSTALL_K3S_EXEC="server \
        --server https://10.10.10.11:6443 \
        --disable-cloud-controller \
        --disable=servicelb \
        --disable=traefik \
        --kubelet-arg=cloud-provider=external \
        --node-ip=10.10.10.13 \
        --flannel-iface=enp7s0 \
        --node-label node-type=master" \
      sh -
  - systemctl start hetzner-vip-sync.service
```

### Step 3.3 - Launch the servers

Run the following from your local machine. Create master 1 first and wait for
K3s to finish bootstrapping before creating masters 2 and 3, because they need
to reach master 1's etcd ports (`2379`/`2380`) during the join.

```bash
# Create master 1 — bootstraps the cluster
hcloud server create \
  --name k8s-master-1 \
  --type cx22 \
  --image ubuntu-24.04 \
  --network k8s-network \
  --ssh-key <your-ssh-key> \
  --user-data-from-file master-1-init.yaml

# Create masters 2 and 3
hcloud server create \
  --name k8s-master-2 \
  --type cx22 \
  --image ubuntu-24.04 \
  --network k8s-network \
  --ssh-key <your-ssh-key> \
  --user-data-from-file master-2-init.yaml

hcloud server create \
  --name k8s-master-3 \
  --type cx22 \
  --image ubuntu-24.04 \
  --network k8s-network \
  --ssh-key <your-ssh-key> \
  --user-data-from-file master-3-init.yaml
```

## Step 5 - Update Your Kubeconfig to Use the VIP

Point your kubeconfig at `10.10.10.10` instead of any individual node IP so
that `kubectl` keeps working even if the current leader is replaced.

On any master node, copy and patch the K3s-generated kubeconfig:

```bash
cp /etc/rancher/k3s/k3s.yaml ~/k3s-vip.yaml
sed -i \
  's|https://127.0.0.1:6443|https://10.10.10.10:6443|g' \
  ~/k3s-vip.yaml
```

Copy `~/k3s-vip.yaml` to your local machine and use it as your `KUBECONFIG`:

```bash
export KUBECONFIG=~/k3s-vip.yaml
kubectl get nodes
```

## Step 6 - Test Failover

We will now simulate a master node failure to verify that Kube-VIP re-elects
a leader and the `hetzner-vip-sync` service updates Hetzner network route
automatically.

### Step 6.1 - Note the current leader

Before starting, record which node currently holds the VIP:

```bash
kubectl -n kube-system get lease kube-vip \
  -o jsonpath='{.spec.holderIdentity}'
```
![image8](images/image8.png)


### Step 6.2 - Simulate a failure

Power off the current leader using Hetzner CLI. A hard power-off is a
more realistic test than a clean reboot, because it forces Kube-VIP to detect
the failure through lease expiry rather than a graceful shutdown:

```bash
hcloud server poweroff k8s-master-1
```

### Step 6.3 - Watch the election on another master

SSH into `k8s-master-2` and tail the sync service log:

```bash
journalctl -u hetzner-vip-sync.service -f
```

Within approximately 10–15 seconds (Kube-VIP lease timeout + one polling
cycle) you should see:

```text
Became leader. Updating private route to server <k8s-master-2-id>...
```
![image9](images/image9.png)

### Step 6.4 - Verify the route and cluster state

Check Hetzner Console 
![image10](images/image10.png)

Verify the cluster is still operational from your local machine:

```bash
kubectl get nodes
```

Expected output (master 1 `NotReady`, masters 2 and 3 still `Ready`):

```text
NAME           STATUS     ROLES                       AGE
k8s-master-1   NotReady   control-plane,etcd,master   15m
k8s-master-2   Ready      control-plane,etcd,master   13m
k8s-master-3   Ready      control-plane,etcd,master   13m
```

API server and etcd remain fully available because etcd requires only a
majority quorum (2 of 3 nodes) to continue operating.

### Step 6.5 - Restore master 1

Power master 1 back on:

```bash
hcloud server poweron k8s-master-1
```

Once it rejoins (`kubectl get nodes` shows it `Ready` again), the
`hetzner-vip-sync` service on master 1 will detect that it does **not** hold
the VIP and remain in follower state. Leadership stays on master 2 until
master 2 itself fails or restarts — Kube-VIP does not preempt a healthy
leader.

## Step 7 (Optional) - Expose the VIP Publicly via a Floating IP

The setup above covers a completely private control plane,
which is perfect for secure intra-cluster communication.
But if you want to run kubectl commands from your local laptop or a CI/CD pipeline outside of Hetzner network,
you can add a public Hetzner Floating IP, same as the private VIP setup.

> Note: Hetzner requires your master nodes to have a Primary Public IPv4 address attached to them in order to route a Floating IP.

If you want to set this up, the process is straightforward — here are the extra steps:

**Provision a Floating IP**: Create an IPv4 Floating IP in Hetzner Cloud Console in the same datacenter as your master nodes.

**Update the Sync Script**: Edit your hetzner-vip-sync.sh script on every master node to include a second API call. Whenever Kube-VIP changes leaders,
the script should also hit the `/v1/floating_ips/<ID>/actions/assign` endpoint to move the Floating IP alongside the private Alias IP.

**Update the TLS SAN**: To prevent kubectl from throwing strict certificate errors, you must add the new public Floating IP to your K3s configuration (via `--tls-san`) and restart K3s to rotate the certificates on all nodes.

**Update your Kubeconfig**: Change the server address in your local kubeconfig file from the private VIP to your new public Floating IP.

## Conclusion

You now have a three-master highly available K3s control plane on Hetzner
Cloud, without a managed Load Balancer. Here's a summary of how the pieces
fit together:

1. **Kube-VIP** performs leader election using a Kubernetes `Lease` object in
   `kube-system`. The elected leader binds the private VIP `10.10.10.10` to
   its `enp7s0` interface. Because Hetzner's SDN does not pass ARP broadcasts,
   this binding is local only — it acts solely as a detectable signal.
2. The **`hetzner-vip-sync`** service on each master polls for that local
   binding every five seconds. When detected, it calls Hetzner API to
   update the private network route so that all traffic to `10.10.10.10`
   inside Hetzner network is forwarded to the new leader. Failover
   completes within one polling cycle.
3. *(Optional)* A Hetzner **Floating IP** can be reassigned to the leader at
   the same time, providing a stable public endpoint for external access.
